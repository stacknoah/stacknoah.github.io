## 1. 결함의 본질
사용자 입력이 "데이터"가 아니라 "쿼리 문법의 일부"로 해석되는 결함. 서버가 입력값을 SQL 문자열에 그대로 이어붙이면, 입력 안의 따옴표나 키워드가 쿼리의 구조 자체를 바꿔버린다.

주입 계열은 다 같은 뿌리를 공유한다 — 데이터와 코드의 경계가 무너져 데이터가 코드로 승격되는 것. 다만 노리는 해석기가 다르다. SQLi는 SQL 파서, NoSQL Injection은 쿼리 객체의 타입·구조, Command Injection은 셸, XSS는 브라우저의 JS 엔진. 이 관점이 서면 새 주입 취약점을 만나도 "어느 해석기로 승격되나"만 물으면 된다.

## 2. 공격 대상
데이터베이스. 결과로 데이터 유출(타 사용자 정보·비번 해시), 인증 우회, 데이터 조작(수정·삭제)이 가능하고, DB 설정에 따라 파일 읽기·쓰기나 RCE까지 확장되기도 한다.

## 3. 전제 조건
- 입력이 쿼리 문자열에 직접 삽입됨 (문자열 조립)
- 파라미터화(Prepared Statement / 바인딩)의 부재
- 입력 검증·이스케이프의 부재

핵심: 값이 들어갈 자리에 문법이 들어갈 수 있으면 성립한다.

## 4. 공격 경로

### 모든 payload의 공통 뼈대
서버가 만드는 쿼리는 대개 `... WHERE uid='[내 입력]' and upw='[내 입력]'` 꼴이다. 내 입력은 작은따옴표 안에 갇혀 있으니, 논리를 바꾸려면 두 가지를 순서대로 한다.

1. 따옴표 탈출: 값 자리에 `'`를 넣어 문자열 밖으로 나온다. (최소 검증 — `'` 하나만 넣어 에러가 나면 주입구가 살아있다는 신호다.)
2. 꼬리 처리: 내 주입 뒤에 남는 원래 쿼리 조각(`' and upw='...'`)을 무력화한다. 주석으로 잘라내거나, 문법이 valid하도록 배치한다.

주석 함정: MySQL에서 `--`는 뒤에 공백이 있어야 주석이다. 그런데 끝 공백은 폼 전송·복붙에서 잘려나가 조용히 실패한다. 그래서 `-- -`(공백 뒤에 문자를 하나 더 붙임)나 `#`을 쓴다. 이게 표준 관용구다.

### 인증 우회
"행이 반환되면 통과"인 로그인류에서 쓴다.

- 특정 유저로 로그인:
  ```
  입력: admin' -- -
  결과: SELECT * FROM users WHERE uid='admin' -- -' and upw='x';
  ```
  uid를 admin으로 닫고 비번 검사를 주석으로 날린다 → 비번 없이 admin으로 인증.

- 조건을 항상 참으로 (전체 행 반환):
  ```
  입력: ' OR '1'='1' -- -
  ```

### UNION 기반 추출 — 다른 테이블에서 꺼내기
결과가 화면에 보이고(in-band), 원하는 데이터가 현재 쿼리의 테이블 밖에 있을 때 쓴다. UNION은 두 번째 SELECT의 행을 원래 결과에 이어붙이므로, 다른 테이블을 SELECT해 붙이면 그 데이터가 화면에 딸려 나온다. 순서가 정해져 있다.

1. 컬럼 개수 맞추기 (UNION은 양쪽 컬럼 수가 같아야 성립):
   ```
   ' ORDER BY 1-- -   정상
   ' ORDER BY 4-- -   정상
   ' ORDER BY 5-- -   에러  → 컬럼은 4개
   ```
   정상으로 뜨는 마지막 숫자가 컬럼 수다.

2. 출력 위치 찾기 (모든 컬럼이 화면에 뜨진 않는다):
   ```
   ' UNION SELECT 1,2,3,4-- -
   ```
   화면에 실제로 보이는 숫자가 "데이터를 실을 수 있는 칸"이다.

3. 스키마를 모르면 이름부터 캔다 (information_schema):
   ```
   현재 DB의 테이블 목록:
   ' UNION SELECT 1,GROUP_CONCAT(table_name),3,4 FROM information_schema.tables WHERE table_schema=database()-- -

   특정 테이블의 컬럼 목록:
   ' UNION SELECT 1,GROUP_CONCAT(column_name),3,4 FROM information_schema.columns WHERE table_name='fake_table_name'-- -
   ```

4. 추출: 출력 위치에 원하는 컬럼을 싣는다. 여러 값을 한 칸에 몰려면 CONCAT(한 행) / GROUP_CONCAT(여러 행).
   ```
   ' UNION SELECT 1,CONCAT(fake_col2,fake_col3,fake_col4),3,4 FROM fake_table_name-- -
   ```
   uid는 빈 값(`'`)으로 시작해 원래 users 행은 안 나오고 UNION 행만 뜨게 한다.

### Blind — 결과가 화면에 안 보일 때
참/거짓만 알 수 있으니 값을 한 글자씩 캐낸다. `substr(값,위치,1)`로 한 글자를 떼어 비교한다.

- Boolean-based (응답 내용이 참/거짓에 따라 달라질 때):
  ```
  admin' AND substr(upw,1,1)='a'-- -    맞으면 로그인 성공 화면, 틀리면 실패
  ```
  위치와 문자를 바꿔가며 한 글자씩 확정한다. 보통 스크립트로 자동화.

- Time-based (응답 내용이 안 변할 때, 지연으로 판정):
  ```
  admin' AND IF(substr(upw,1,1)='a', sleep(3), 0)-- -    3초 걸리면 참
  ```

### Error-based / Stacked
- Error-based: DB 오류 메시지가 화면에 노출될 때, 데이터를 오류 메시지에 실어 뽑아낸다.
- Stacked queries: `;`로 완전히 별도의 쿼리를 실행한다. 드라이버가 다중 쿼리를 허용할 때만 된다(일부에 한정).

## 5. 판별점
- 입력이 SQL 쿼리에 문자열로 들어가나? (source → 쿼리라는 sink)
- 파라미터 바인딩이 아니라 문자열 연결(f-string, `+`, `%`)인가?
- 따옴표·`OR`·주석(`--`)을 넣으면 응답이나 오류가 달라지나?

셋 다 예 → SQLi 가능. 그리고 결과가 화면에 보이나로 in-band(UNION)와 blind(한 글자씩)가 갈린다.

## 6. 방어
- Prepared Statement (파라미터 바인딩): 쿼리 구조를 먼저 고정하고 값은 따로 전달한다. 입력이 절대 문법으로 승격되지 못하므로 근본 해법이다.
- 입력 검증 / 화이트리스트: 바인딩이 안 되는 부분(테이블·컬럼명, 정렬 방향)은 화이트리스트로 막는다.
- 최소 권한 DB 계정: 뚫려도 피해 범위를 줄인다.
- ORM: 보통 내부적으로 파라미터화하지만, raw 쿼리를 섞어 쓰면 그대로 무력화된다.

이스케이프(따옴표 치환)만으로 막으려는 건 취약하다. 근본은 데이터와 코드의 분리, 즉 파라미터화다.
